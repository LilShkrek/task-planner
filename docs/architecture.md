# Архитектура проекта

## Компоненты

### Go backend

Отвечает за:

- REST API для пользовательских задач;
- базовую валидацию входных данных;
- сохранение задач, планов и истории выполнения;
- вызов Python ML service;
- финальную сборку плана выполнения.

### Python ML service

Отвечает за:

- извлечение признаков из текста и параметров задачи;
- загрузку методов тайм-менеджмента и шаблонов планов из PostgreSQL;
- рекуррентную обработку последовательности токенов через PyTorch `nn.GRU`;
- dense/perceptron-блок выбора метода тайм-менеджмента;
- расчет scores по методам и параметров планирования;
- генерацию чернового плана;
- расчет рекомендаций по расписанию.

Текущая реализация учебная: используется настоящая архитектура `torch.nn.Module` и минимальный training pipeline на небольшом JSON-датасете. Если файл обученных весов существует, ML service загружает его при создании модели. Если файла нет, используется фиксированная начальная инициализация.

Training pipeline:

1. Скрипт `python -m app.training.train` загружает методы тайм-менеджмента из PostgreSQL.
2. Учебные примеры читаются из `ml-service/data/training_tasks.json`.
3. Для каждого примера строятся признаки задачи и целевой индекс метода.
4. Модель обучается по `CrossEntropyLoss` для метода и небольшой MSE-добавке для параметров планирования.
5. Веса сохраняются в `ml-service/artifacts/time_management_model.pt`.

Конвейер ML service:

1. Текст задачи разбивается на токены.
2. Каждый токен превращается в небольшой вектор признаков.
3. Из PostgreSQL загружается каталог методов и последний шаблон плана для каждого метода.
4. Модель создается с выходным слоем под количество методов из БД и пытается загрузить веса из `MODEL_PATH`.
5. Последовательность признаков проходит через `GRU`.
6. Скрытое состояние `GRU` объединяется с числовыми признаками задачи: приоритет, оценка времени, дедлайн, сложность текста.
7. Dense/perceptron-блок формирует:
   - `scores` по методам тайм-менеджмента;
   - `planning_params`: длительность фокус-сессии, перерыва, число блоков, резерв на проверку.
8. По максимальному score выбирается метод из загруженного каталога.
9. Модуль генерации плана берет JSON-шаблон выбранного метода из БД и строит черновые шаги.
10. Модуль планирования берет `schedule_hint` из того же JSON-шаблона и подставляет параметры модели.

### PostgreSQL

Хранит:

- пользователей;
- задачи;
- методы тайм-менеджмента;
- шаблоны планов;
- планы и шаги планов;
- историю выполнения.

## Основной сценарий

1. Пользователь отправляет задачу в Go backend.
2. Backend валидирует и сохраняет задачу.
3. Backend отправляет задачу в ML service.
4. ML service читает методы и шаблоны из PostgreSQL.
5. ML service возвращает выбранный метод, scores, уверенность, параметры планирования, черновые шаги и подсказки по расписанию.
6. Backend формирует итоговый план и сохраняет его.
7. Backend возвращает пользователю задачу, рекомендацию и план.
