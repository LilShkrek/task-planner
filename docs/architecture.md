# Архитектура проекта

## Компоненты

### Go backend

Отвечает за:

- REST API для пользовательских задач;
- базовую валидацию входных данных;
- сохранение задач, планов и истории выполнения;
- вызов Python ML service;
- финальную сборку плана выполнения.

Вызов ML service выполняется с отдельным настраиваемым таймаутом `ML_SERVICE_TIMEOUT_SECONDS`. Для локальной генеративной модели значение по умолчанию увеличено до 90 секунд, потому что CPU inference `cointegrated/rut5-base-multitask` может быть заметно дольше старой заглушки. Если таймаут истек, backend возвращает пользователю `504 Gateway Timeout`, а не общий `502`.

### Python ML service

Отвечает за:

- извлечение признаков из текста и параметров задачи;
- загрузку каталога из 40 методов тайм-менеджмента и шаблонов планов из PostgreSQL;
- рекуррентную обработку последовательности токенов через PyTorch `nn.GRU`;
- dense/perceptron-блок оценки методов тайм-менеджмента;
- расчет scores по методам и параметров планирования;
- выбор комбинации из 3-5 методов на основе scores, групп и ролей;
- генерацию итоговых текстов плана локальной предобученной text-to-text моделью.

Текущая реализация учебная:

- Модель выбора метода: локальная `GRU + dense/perceptron`.
- GRU/dense-модель возвращает scores по всем методам, после чего отдельный модуль выбирает 3-5 методов для комбинированного плана.
- Генеративная модель ответа: предобученная `cointegrated/rut5-base-multitask`, запущенная локально через `transformers`.
- Генерация настроена на устойчивый локальный запуск: по умолчанию используется greedy decoding (`GENERATIVE_NUM_BEAMS=1`), ограничение входа `GENERATIVE_MAX_INPUT_TOKENS=384` и ограничение новых токенов `GENERATIVE_MAX_NEW_TOKENS=64`.

Если файл обученных весов для модели выбора метода существует, ML service загружает его. Если файла нет, используется фиксированная начальная инициализация. Генеративная модель скачивается при сборке Docker-образа и хранится в локальном cache.

Training pipeline:

1. Скрипт `python -m app.training.train` загружает методы тайм-менеджмента из PostgreSQL.
2. Учебные примеры читаются из `ml-service/data/training_tasks.json`.
3. Для каждого примера строятся признаки задачи и целевой индекс метода.
4. Модель обучается по `CrossEntropyLoss` для метода и небольшой MSE-добавке для параметров планирования.
5. Веса сохраняются в `ml-service/artifacts/time_management_model.pt`.

Конвейер ML service:

1. Текст задачи разбивается на токены.
2. Каждый токен превращается в небольшой вектор признаков.
3. Из PostgreSQL загружается каталог методов и последний шаблон плана для каждого метода.
4. Модель создается с выходным слоем под количество методов из БД и пытается загрузить веса из `MODEL_PATH`.
5. Последовательность признаков проходит через `GRU`.
6. Скрытое состояние `GRU` объединяется с числовыми признаками задачи: приоритет, оценка времени, дедлайн, сложность текста.
7. Dense/perceptron-блок формирует:
   - `scores` по методам тайм-менеджмента;
   - `planning_params`: длительность фокус-сессии, перерыва, число блоков, резерв на проверку.
8. Методы ранжируются по score, затем multi-method selection выбирает 3-5 методов с разными группами и ролями.
9. Из шаблонов выбранных методов БД собирается комбинированный этапный шаблон: цель, приоритизация, декомпозиция, выполнение/распределение времени, контроль.
10. Генеративный модуль получает `title`, `description`, `context`, выбранную комбинацию методов, `planning_params`, semantic structure и комбинированный шаблон.
11. Предобученная text-to-text модель формирует `summary`, `schedule_hint`, заголовки и описания шагов.
12. Количество шагов и порядок этапов остаются под контролем шаблонов из БД и selection-модуля.

Генеративный модуль не заменяет выбор методов. Он генерирует только тексты ответа, а шаблоны БД и selection-модуль задают количество шагов, порядок шагов и общую форму плана.

Каталог методов хранится в таблице `time_management_methods`. У метода есть поля `code`, `name`, `description`, `best_for`, `method_group`, `role`. В каталоге 40 методов, сгруппированных по этапам планирования: формулировка цели, приоритизация, декомпозиция, распределение времени, старт / борьба с прокрастинацией, организация потока задач, выполнение, контроль / завершение. Текущий pipeline возвращает `ranked_methods`, `selected_methods` и `explanation`; поле `method_code` сохранено как основной метод для совместимости с backend и таблицей `plans`.

Если backend или внешний клиент закрывает соединение до завершения генерации, ML service перехватывает `BrokenPipeError`/`ConnectionResetError` при отправке JSON и пишет короткое диагностическое сообщение без traceback.

### PostgreSQL

Хранит:

- пользователей;
- задачи;
- методы тайм-менеджмента;
- шаблоны планов;
- планы и шаги планов;
- историю выполнения.

## Основной сценарий

1. Пользователь отправляет задачу в Go backend.
2. Backend валидирует и сохраняет задачу.
3. Backend отправляет задачу в ML service.
4. ML service читает методы и шаблоны из PostgreSQL.
5. ML service возвращает основной метод, scores, `ranked_methods`, `selected_methods`, объяснение комбинации, параметры планирования, summary, шаги и подсказку по расписанию.
6. Backend формирует итоговый план и сохраняет его.
7. Backend возвращает пользователю задачу, рекомендацию и план.
